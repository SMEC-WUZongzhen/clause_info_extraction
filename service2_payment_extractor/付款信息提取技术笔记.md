# 付款信息提取技术笔记

> **项目路径**: `service2_payment_extractor`
> **基座模型**: Qwen3.5-9B（SFT + RFT 微调）
> **技术栈**: Python 3.12 / FastAPI / LangGraph / Milvus / BM25 / BGE-Reranker
> **最后更新**: 2026-06-16

---

## 1. 项目概述

### 1.1 业务背景

电梯设备采购合同中包含大量付款条款，涉及设备款、安装款两条线，每条线有十余种标准付款节点（预付款、提货款、货到工地、公司验收后、质保金等）。人工录入 SIS 系统耗时且易错，本服务通过自研微调大模型 + RAG 检索增强 + 多阶段工程兜底，实现从合同原文到结构化付款节点的端到端自动提取。

### 1.2 核心能力

| 能力 | 说明 |
|------|------|
| 付款节点提取 | 从合同条款段落中提取付款类型、比例、金额，映射至 13 种设备节点 / 12 种安装节点 |
| 质保信息提取 | 提取质保期、生效条件、终止条件 |
| 付款时效提取 | 提取付款天数、最迟付款节点、最迟付款时间 |
| 混签条款处理 | 自动判定混签条款归属（设备/安装/双轨），展开双副本独立处理 |
| 提取结果比对 | 与 SIS 系统标准答案做 F1 / 精确率 / 召回率评估 |
| 合同金额比对 | LLM 抽取合同总金额，与 SIS 记录做一致性校验 |

### 1.3 输入输出

```
输入: { id, paragraphs[{ clause, clause_context, clause_class }], operation_type, sis_payment_stages? }
输出: ExtractionResponse | AnalysisResponse
```

- `clause`: 原子子条款原文（如"合同签订后7日内支付30%预付款"）
- `clause_context`: 子条款所在段落的完整上下文，用于语义增强检索
- `clause_class`: 条款分类标签（设备付款/安装付款/混签付款/质保期）

## 2. 系统架构

### 2.1 整体架构

系统基于 **LangGraph StateGraph** 构建有状态工作流，核心节点如下：

```
payment_info_extractor → route_after_extraction → comparison_node (analyze模式)
                                           ↓
                                      output_node → aggregator → END
```

- `payment_info_extractor_node`: 核心提取节点，包含 12 阶段流水线（约 1963 行）
- `comparison_node`: 仅 `analyze` 模式触发，将提取结果与 `sis_payment_stages` 做模糊匹配评估
- `output_node`: 打包提取结果 + 比对结果为最终输出
- `aggregator_node`: 收集 `final_output` 并返回 `AIMessage`

### 2.2 双轨并行处理

合同付款条款分为 **设备付款（equipment_payment）** 和 **安装付款（installation_payment）** 两条链路：

- 独立的 RAG 检索库（Milvus collection + BM25 语料库分别构建）
- 独立的标准节点映射表（设备 13 节点 / 安装 12 节点）
- 独立的批量复核流程（Stage 8 设备侧 / 安装侧并行 swimlane）
- 独立的去重校验规则

混签条款（`clause_class=["混签付款条款"]`）经 LLM 判定后，仅在 `both` 时展开为双轨副本，避免对单归属条款的无谓重复提取。

## 3. 12 阶段提取流水线

核心节点 `payment_info_extractor_node`（`app/agents/payment_info_extractor_node.py:888`）实现了 12 阶段流水线，以下是各阶段的详细说明。

### Stage 1 — 预过滤（prefilter）

**目标**: 在 LLM 调用前用硬规则剔除明显非付款条款，节省 LLM 调用成本。

三层过滤：
1. **关键词过滤**: 含"违约金/罚款/赔偿损失/质保期保养预留费"等关键词的条款直接剔除。关键词列表通过 `.env` 的 `CLAUSE_FILTER_KEYWORDS` 配置，无需改代码。
2. **协商拒绝过滤**: 检查 `clause_context` 是否包含"不予调整"等拒绝协商的上下文，此类条款不具备实际付款约束力。
3. **表格/碎片/备注行过滤**: `_is_table_or_non_clause` 通过正则识别表格分隔符、空碎片、仅注释行等非条款文本。

### Stage 2 — 条款有效性校验（validate）

**目标**: 对每条条款做 LLM 有效性判断，fail-closed 策略（仅 `is_valid=True` 才放行）。

**并发执行**: `validate_payment_clause_single` 对每条条款独立发起 LLM 调用，`asyncio.gather` 并发。

**三级兜底**:
- LLM 返回 `True/False` → 直接使用
- LLM 返回 `None`（无法判定）→ `_should_force_valid_by_text_features`：
  - 要求同时满足：长度 ≥ 30 字符 + 含 `%` 或 `％` + 含付款动作关键词（支付/付款/付清/结算/定金/预付/提货/货到/验收/移交/进度款/尾款/余款/请款/汇入/汇至/到货款）
- LLM 返回 `False` → `_should_force_valid`（更强硬的白名单兜底）：
  - 读取 `business_dict.force_valid` 配置：`min_clause_len`(20)、`node_pct_gap`(20)、`node_keywords`(18 词)、`exclude_keywords`(12 词)
  - 排除含排除关键词的条款
  - **双向窗口扫描**: 找到所有 `\d+(\.\d+)?\s*[%％]` 位置，对每个百分比检查 ±(gap + max_kw_len) 字符范围内是否出现节点关键词。同时支持"关键词在前"（提货款 95%）和"百分比在前"（5%质保金）两种模式。

### Stage 3 — 混签归属路由（resolve_mixed）

**目标**: 对 `clause_class=["混签付款条款"]` 的条款，通过 LLM 判定归属。

- `classify_mixed_category_single` 输出 `equipment_payment` / `installation_payment` / `both`
- `both` → 展开为设备 + 安装双副本，各自独立进入后续流程
- 残留混签条款强制展开为双轨（防止 LLM 漏判）

### Stage 4 — 质保信息提取（warranty）

**目标**: 独立提取质保期信息，与付款条款提取并行。

`extract_summary_warranty` 提取：
- `warranty`: 质保期时长（如"24个月"）
- `effective_conditions`: 质保生效条件
- `closed_end_conditions`: 质保终止条件

> 设计约束 H8: 质保信息仅通过函数返回值传递，不污染 State，避免对后续付款提取造成干扰。

### Stage 5 — 并发 RAG + LLM 初提取（concurrent_extract）

**目标**: 对每条付款条款并行执行 RAG 检索 + LLM 结构化提取。

**核心流程**:
1. 对每条段落创建 `asyncio.create_task`，`asyncio.gather` 并发执行 `_process_single_payment_paragraph`
2. 每条段落内部：
   - RAG 检索相似条款作为 Few-shot 示例参考
   - LLM 结构化提取：`payment_type`、`payment_ratio`、`payment_amount`
   - JSON 解析三级降级（详见第 6 节）
   - `ratio` 兜底：若 LLM 未输出比例但原文含 `\d+%`，正则提取
3. 失败率 adjudication: `get_failure_rate_threshold`（默认 0.5），超过阈值则设置 `extraction_partial=True`

**辅助规则**: `_should_strip_ratio` — 条款含辅助费用关键词（保养费/指导费/免保期保养费）且无显式百分比时，强制 `ratio=None`，防止将非付款比例误识别。

### Stage 6 — 严格去重（strict_dedupe）

**目标**: 对初提取结果做字符串相似度去重，减少后续 LLM 批量复核的输入量。

`_remove_duplicate_payment_items`（`payment_info_extractor_node.py:403`）:

**两层阈值**:
- 严格阈值 = 0.95（`DEDUPE_ITEM_SIM_STRICT`）— Stage 6 使用
- 宽松阈值 = 0.80（`DEDUPE_ITEM_SIM_LOOSE`）— 默认/其他场景

**去重守卫**（跳过去重的条件）:
1. `clause_class` 不同（混签展开的双轨副本必须保留）
2. `payment_type` 不同（同一条款拆分出的多个子节点必须保留）

**重复判定**（优先级递降）:
1. 同 `payment_type` + 同规范化 `amount` → 重复
2. 全文包含关系（`text1 in text2`）→ 重复
3. `SequenceMatcher.ratio() >= threshold` → 重复

**胜出者选择**: 优先保留含证据的条款（原文含金额字符串或 `\d+%` 模式），其次取更长文本。

### Stage 7 — DSU 上下文合并（merge_contexts）

**目标**: 对相似/重叠的条款上下文做聚类合并，减少 LLM 批量复核时的输入冗余。

**数据结构**: `_DSU` 类（`payment_info_extractor_node.py:251`）实现并查集，含路径压缩 + 按秩合并。

**合并算法** `_deduplicate_and_merge_contexts`（`payment_info_extractor_node.py:276`）:

1. **初始分组**: 相同上下文文本归入同一组
2. **O(n²) 两两扫描**: 满足以下任一条件则 `union`:
   - 子串包含关系（`ctx_i in ctx_j`）
   - 前后缀重叠 ≥ 20 字符（`_CONTEXT_OVERLAP_MIN_LEN`）且 `SequenceMatcher.ratio() >= 0.85`（`_CONTEXT_MERGE_SIM_THRESHOLD`）
3. **结果聚合**: 按 DSU 根分组，组内按最小原始索引排序，通过 `_merge_overlapping_strings` 合并代表文本（取超集；无重叠时取更长文本）

设备组与安装组分别独立执行此流程。

### Stage 8 — 批量复核（summary_review）

**目标**: 将初提取结果与合并后的上下文一并送入 LLM 做批量交叉校验。

`extract_summary` 将设备侧 + 安装侧的所有条目与合并后的上下文交叉送入 LLM，LLM 可：
- 确认 / 修正 `payment_type`、`ratio`、`amount`
- 剔除无效节点
- 补充遗漏节点

**9B 模型适配**: 使用 `prefixed_key`（`eq_N` / `in_N`）编号，避免 9B 模型在批量输入时重新编号导致错位。

**金额幻觉防护**: `_amount_appears_in_text` — LLM 返回的 `final_amount` 必须在条款原文中出现（容忍千分位、`.0` 后缀、万元/万单位转换），否则不接受该金额覆盖。

**设备侧复核 swimlane**（6 步）:
1. 无效节点过滤
2. 比例计算（"支付至 X%" → 差额比例）
3. 金额核算
4. 合并去重
5. 映射 13 类标准节点
6. 字段校验

**安装侧复核 swimlane**（3 步）:
1. 0 元条款过滤
2. 质保金合并
3. 白名单映射 12 类 + 字段校验

### Stage 9 — 单组去重复核（dedupe_review）

**目标**: 对同一 `(clause_category, payment_type)` 下存在多条记录的分组，用 LLM 选最优 ID。

`_validate_extraction_results`（`payment_info_extractor_node.py:577`）:
1. 硬编码检测重复组
2. 对每个重复组并发调用 `verify_single_group_single`，LLM 从候选中选择最优条目
3. 代码兜底 `_enforce_unique_payment_type`

### Stage 10 — 结果验证与回填（result_verify）

**目标**: 硬编码比对 + LLM 验证 + 代码强制唯一，确保最终结果可靠。

`_enforce_unique_payment_type`（`payment_info_extractor_node.py:519`）:
- 按 `(clause_category, payment_type)` 分组
- 多条记录的分组 → `_pick_best_clause` 选胜出者
- 评分逻辑: `(score_action, score_len)`
  - `score_action`: 2 = 含付款动词 + 含金额/比例；1 = 仅含动词；0 = 均无
  - `score_len`: 条款原文长度

### Stage 11 — 未匹配兜底恢复（recovery_fallback）

**目标**: 恢复在批量复核中被错误丢弃的条款。

从初提取结果中找回未被 summary_review 处理的 key，施加以下守卫：
- 相似度 ≥ 0.85 的冲突项跳过
- 0 元条款跳过
- 高相似重复跳过
- `payment_type` 守卫

### Stage 12 — 算术校核与清理（postprocess）

`_postprocess_final_payment_infos`（`payment_info_extractor_node.py:171`）:

1. **算术推断**: 条款无 `%` + 上下文中存在唯一总额 + `amount == total`（±0.5）→ 强制 `ratio = 1.0`
2. **零金额清理**: `amount ∈ {"0", "0.0", "0.00", "0.000", ""}` → 丢弃
3. **空壳清理**: `ratio` 为 None/0 且 `amount` 为 None/0 → 丢弃

### 附加 — 付款时效提取

在 12 阶段完成后执行（`payment_info_extractor_node.py:1860`）:

1. **special_clause_content**: 所有唯一 `payment_clause` 文本以 `\n` 拼接（文档级汇总）
2. **并发提取**: 对唯一 `(clause, context, category)` 组合并发调用 `extract_payment_timing_single`，提取:
   - `payment_days`: 付款天数
   - `latest_payment_stage`: 最迟付款节点
   - `latest_payment_date`: 最迟付款时间（天数）
3. **硬门控回写**: `latest_payment_stage` / `latest_payment_date` 仅当 `payment_clause` 原文包含"最迟"二字时才保留，否则强制置 null。防止质保条款中的"最迟不超过..."污染付款时效字段。
4. **类别白名单**: `latest_payment_stage` 必须在当前 `clause_category` 的标准节点白名单内，否则置 null。

## 4. RAG 检索系统

### 4.1 三阶段混合检索架构

`app/utils/rag_retriever.py` 实现了 BM25 + Milvus 向量 + BGE-Reranker 的三阶段检索:

```
输入条款 → Phase 1: 并行双路召回 → Phase 2: 融合去重 → Phase 3: Cross-Encoder 精排 → Top-K 结果
```

### 4.2 Phase 1 — 并行双路召回

BM25 和 Milvus 向量检索通过 `asyncio.gather` 并发执行:

**BM25 召回**:
- 使用 `rank_bm25.BM25Okapi`，`jieba.cut` 中文分词
- 设备/安装分别构建独立语料库（pickle 文件从 BOS 下载，SHA256 完整性校验）
- 最低分阈值: `RAG_BM25_MIN_SCORE`（默认 20）

**Milvus 向量召回**:
- 使用 `pymilvus.MilvusClient`，设备/安装分别使用不同 collection
- Embedding: 远程 Qianfan `qwen3-embedding-0.6b` API，httpx 异步调用
- L2 归一化后检索，最低距离阈值: `RAG_VECTOR_MIN_DISTANCE`（默认 0.5）
- Embedding 并发控制: `asyncio.Semaphore`（默认 4），指数退避重试
- 失败时降级为零向量（不阻断流程）

### 4.3 Phase 2 — 融合去重

- 向量结果优先（语义更准确），BM25 结果补充（关键词覆盖）
- 按 `text` / `clause_text` 字段去重

### 4.4 Phase 3 — Rerank 精排

- 模型: `BAAI/bge-reranker-large`（本地加载，`transformers.AutoModelForSequenceClassification`）
- Cross-Encoder 方式: 对每个 `(query, candidate_text)` pair 打分，`torch.no_grad()`
- 按分数降序排列，取 Top-K

### 4.5 并发与连接管理

- `httpx.AsyncClient` 按 event loop 管理（`weakref.WeakKeyDictionary`），防止跨事件循环错误
- Embedding API 调用使用独立信号量控制并发（默认 4）
- 指数退避重试机制

## 5. LLM 微调与调用

### 5.1 模型微调

基座模型为 **Qwen3.5-9B**，通过两阶段微调适配付款条款提取任务:

| 阶段 | 方法 | 目的 |
|------|------|------|
| SFT | 监督微调 | 用标注的付款条款-结构化结果对训练，让模型学会从条款文本提取 `payment_type` / `ratio` / `amount` |
| RFT | 强化微调 | 基于正确/错误提取结果的反馈信号进一步优化，提升边界 case 的判断能力 |

微调后的模型在特定领域抽取准确率上显著优于通用基座模型。

### 5.2 LLM 调用场景

| 场景 | 函数 | 说明 |
|------|------|------|
| 条款有效性校验 | `validate_payment_clause_single` | 单条款并发，fail-closed |
| 混签归属判定 | `classify_mixed_category_single` | 单条款并发 |
| 付款节点初提取 | `extract_payment_info` | RAG 辅助，单条款并发 |
| 批量复核 | `extract_summary` | 设备/安装分组批量送入 |
| 单组去重 | `verify_single_group_single` | 按重复组并发 |
| 付款时效提取 | `extract_payment_timing_single` | 按唯一条款并发 |
| 合同金额抽取 | `contract_price_comparator` | 单次调用 |

### 5.3 LLM 配置

- 支持独立配置付款比例提取的 LLM 参数（`PAYMENT_RATIO_LLM_*` 环境变量），降级到主 LLM 配置
- Token 计数: `tiktoken`（含降级方案）
- 整体请求超时: 默认 300s，超时返回 504

## 6. JSON 解析三级降级

LLM 输出的 JSON 可能格式不合规，`payment_ratio_extractor.py` 实现了三级解析降级:

| 级别 | 方法 | 说明 |
|------|------|------|
| 1 | `json.loads()` | 标准 JSON 解析 |
| 2 | `JSONDecoder` 滑窗扫描 | 在原始文本中滑动寻找 `[...]` 块，逐个尝试解析 |
| 3 | 正则兜底 | 直接用 `\d+%` 正则提取比例，金额用数字模式匹配 |

此外，Stage 3 JSON 解析新增了**悬挂逗号修复**和**对象抢救**逻辑，进一步提升解析鲁棒性。

## 7. 多层去重与兜底体系

系统设计了 7 层去重/兜底策略，从代码层到 LLM 层逐级保障结果质量:

| 层级 | 阶段 | 方法 | 策略 |
|------|------|------|------|
| L1 | Stage 1 | 关键词预过滤 | 硬编码关键词 + `.env` 配置 |
| L2 | Stage 1 | 协商拒绝过滤 | 上下文含"不予调整"等拒绝词 |
| L3 | Stage 6 | 字符串相似度去重 | SequenceMatcher ≥ 0.95，同类型同金额/包含关系 |
| L4 | Stage 7 | DSU 上下文合并 | 并查集聚类，相似度 ≥ 0.85 + 前后缀重叠 ≥ 20 字符 |
| L5 | Stage 9 | LLM 去重复核 | 按重复组并发，LLM 选最优 ID |
| L6 | Stage 10 | 代码强制唯一 | `_enforce_unique_payment_type`，按 `(category, type)` 保证唯一 |
| L7 | Stage 11 | 未匹配恢复 | 从初提取找回被丢弃的有效条款 |

**设计理念**: 单点 LLM 依赖不可靠，通过"代码兜底 + LLM 校验 + 代码再兜底"的交替策略，将系统鲁棒性从模型能力依赖转化为工程化容错体系。

## 8. 异步并发设计

### 8.1 全链路异步

整个流水线基于 `asyncio` 实现，关键并发点:

| 位置 | 并发方式 | 说明 |
|------|----------|------|
| Stage 2 有效性校验 | `asyncio.gather` | 每条条款独立 LLM 调用 |
| Stage 3 混签判定 | `asyncio.gather` | 每条混签条款独立 LLM 调用 |
| Stage 5 初提取 | `asyncio.create_task` + `gather` | 每条段落独立 RAG + LLM |
| Stage 5 内部 RAG | `asyncio.gather` | BM25 + Milvus 并行召回 |
| Stage 8 批量复核 | 双轨并行 | 设备侧 + 安装侧独立 swimlane |
| Stage 9 去重复核 | `asyncio.gather` | 按重复组并发 |
| 付款时效提取 | `asyncio.gather` | 按唯一条款并发 |
| Embedding API | `asyncio.Semaphore` | 并发控制（默认 4） |

### 8.2 并发控制

`app/utils/concurrency.py` 实现了按事件循环的信号量管理:
- `PerLoopSemaphore`: 每个事件循环独立的信号量实例，避免跨循环错误
- LLM 调用信号量: 控制并发 LLM 请求数，防止过载
- Rerank 信号量: 控制 GPU 推理并发

### 8.3 启动预热

Lifespan 中预热以下组件，避免首次请求冷启动:
- `tiktoken` 编码器
- RAG 检索器（BM25 语料库 + Milvus 连接）
- BGE-Reranker 模型

## 9. API 设计

### 9.1 接口列表

| 端点 | 方法 | 功能 |
|------|------|------|
| `/health` | GET | 健康检查 |
| `/extract_payment_info` | POST | 付款信息提取（extract / analyze 双模式） |
| `/compare_contract_price` | POST | 合同总金额抽取 + SIS 对比 |
| `/v1/chat/completions` | POST | OpenAI 兼容接口 |

### 9.2 extract vs analyze 模式

- **extract**: 仅提取付款信息，返回 `extraction_result`
- **analyze**: 提取 + 与 `sis_payment_stages` 比对，额外返回 `correct_payments`、`missed_payments`、`false_payments`、`evaluation_metrics`

### 9.3 评估指标

```python
precision = correct / (correct + false)
recall    = correct / (correct + missed)
f1        = 2 * precision * recall / (precision + recall)
accuracy  = f1  # 当前实现中 accuracy 等于 f1
```

比对采用模糊匹配（`comparison_helper.py`），支持三种匹配状态:
- `fully_matched`: 节点 + 数据完全匹配
- `node_matched_data_mismatch`: 节点匹配但比例/金额不一致
- `node_mismatch`: 节点不匹配

### 9.4 合同金额比对

`/compare_contract_price` 接口:
- LLM 从合同总价条款抽取金额（提示词 `CONTRACT_PRICE_EXTRACTION_PROMPT`）
- 与 `sis_contract_price` 做硬编码差值比对（阈值 `10.0` 元，`_PRICE_DIFF_TOLERANCE`）
- `sis_contract_price` 未传时仅抽取不比对，`comparison_result` 返回 `null`

## 12. 关键设计决策

### 12.1 为什么用 SFT + RFT 而非纯 Prompt Engineering

- 付款条款提取需要理解合同语言的细微差异（如"支付至 90%"与"支付 90%"的语义区别）
- 9B 模型经微调后，在特定领域的结构化提取能力接近大模型，但推理成本显著更低
- RFT 进一步校正 SFT 在边界 case 上的错误判断

### 12.2 为什么用三阶段 RAG 而非纯向量检索

- BM25 擅长关键词精确匹配（如"质保金"、"提货款"等术语）
- 向量检索擅长语义相似（如"验收合格后付款"与"验收通过后支付"）
- Cross-Encoder Rerank 精排弥补双路召回的精度不足
- 三阶段组合在召回率和精确率上均优于单一检索方式

### 12.3 为什么设计 7 层去重

- 9B 模型在批量输入时容易产生重复或编号错位
- 单点 LLM 判断不可靠（可能将同一条款拆分为多个节点）
- 代码兜底是最后一道防线，确保 `(category, payment_type)` 唯一性不受 LLM 波动影响

### 12.4 为什么用并查集合并上下文

- 合同条款经常出现上下文重叠（同一段落被拆分为多条子条款）
- O(n²) 两两比较 + DSU 合并的复杂度可控
- 合并后减少 LLM 批量输入的冗余，降低 token 消耗和编号错位风险

### 12.5 为什么对"最迟"做硬门控

- 质保条款中常出现"最迟不超过..."等表述，但并非付款时效
- 仅当条款原文显式包含"最迟"二字时才保留 `latest_payment_stage` / `latest_payment_date`
- 这是代码规则覆盖 LLM 输出的典型案例：用确定性规则消除模型的语义混淆
