# 模板名：项目接口功能检查（API Functionality Audit）

## 适用场景

拿到后端服务的源码、OpenAPI 文档或接口说明，需要系统化审查若干接口的功能完整性、契约一致性、异常路径覆盖、可观测性与基础安全。适用于任意 Web API 项目。

---

## 前置条件

**环境层**

- [ ] 项目根存在 `README.md` 或 `AGENTS.md`，说明技术栈与运行方式
- [ ] 已明确外部依赖（DB / LLM / 第三方 HTTP / MQ）的调用方式、超时与限流约束

**材料层**（开启对话前必须 @ 或粘贴）

- [ ] 路由 / Controller 文件（含目标接口的实现代码）
- [ ] 请求 / 响应 Schema 文件（BaseModel / DTO / OpenAPI YAML）—— 路由文件较大时单独 `@`
- [ ] 目标接口清单：`method + path + 一句话业务简介`，逐行列出

---

## 标准 Prompt

```
你是熟悉【技术栈】的资深后端 / API 架构师，精通 RESTful、OpenAPI、
JSON Schema / Pydantic、错误处理、可观测性与基础 Web 安全，
正在对一组接口做功能检查。

【技术栈】
<语言 / 框架 / 版本，例：Python 3.11 + FastAPI 0.110 + Pydantic v2>

【接口清单】
<method + path + 一句话简介，逐行列出>

【路由代码】
@<routes/ 或 controller/ 文件1> @<文件2>

【请求 / 响应 Schema】
@<schema/ 或 models/ 文件>（或直接粘贴关键 BaseModel / DTO 定义）

【依赖与外部调用】
<DB、缓存、LLM、第三方 HTTP、消息队列等；注明同步 / 异步、是否有超时与重试>

【已知约束】
<鉴权方式、入参上限、单请求超时、限流策略、幂等要求、SLA 等；不填写则视为无额外约束>

【检查重点】（多选；留空则视为全选）
契约一致性 / 功能完整性 / 异常与边界 / 可观测性 / 安全 / 性能

---
⚠️ 硬约束（每步都须遵守）
· 只引用源码、Schema、文档中实际出现的函数、字段、路由名；不确定时主动发问，不要脑补。
· 严格区分「已确认问题」与「疑似问题」；疑似项必须标注所需额外证据。
· 不输出主观赞美或冗余总结。
· 涉及 token / 个人信息的示例一律脱敏。
---

请严格按以下步骤分析，每步完成后停下，等我确认再继续：

步骤 1 — 契约复述与澄清
复述每个接口的职责与输入输出契约，仅基于实际代码 / Schema / 文档；
遇到歧义或缺失，列出待澄清问题后停下。

步骤 2 — 五维度检查清单（针对每个接口分别输出）
· 契约一致性：路由签名 vs Schema vs 文档的字段、类型、必填 / 可选、默认值、枚举值
  是否一致；HTTP 状态码语义是否正确。
· 功能完整性：是否覆盖所有声明分支；是否有声明未实现、或实现未在文档体现的能力。
· 异常与边界：入参上限、空值 / 超长 / 非法编码、超时、并发、幂等、错误码与错误信息
  是否人机友好；LLM / RAG 接口额外关注重试与 token 计费。
· 可观测性：关键路径日志、trace / request_id、指标、慢请求记录；敏感信息是否脱敏。
· 安全：鉴权与越权、注入风险（SQL / 命令 / 路径 / SSRF）、文件上传与路径穿越、依赖供应链。

步骤 3 — 问题清单
按「严重度 × 置信度」排序（Blocker / Major / Minor × 已确认 / 疑似），每条包含：
· 定位：file:line
· 现象：简述
· 潜在影响：业务 / 安全 / 稳定性后果
· 复核方法：最小复现步骤 / curl 示例 / 单测建议

步骤 4 — 修复方案（等我从步骤 3 挑选要修复的项后再执行）
给出修复方案与改动文件清单；本步骤之前不要改任何代码。
```

## 输入示例

> 提供 3 个不同技术栈、不同复杂度的输入示例，使用时按需替换槽位即可。

### 示例 1 — LLM / RAG 服务（FastAPI + LangGraph）

```
【技术栈】
Python 3.11 + FastAPI 0.110 + Pydantic v2 + LangGraph

【接口清单】
POST /extract_payment_info  接收合同条款段落，返回付款节点提取结果
GET  /health                健康检查

【路由代码】
@service2_payment_extractor/app/api.py

【请求 / 响应 Schema】
@service2_payment_extractor/app/api.py
（ExtractRequest / ExtractResponse / Paragraph 均在同一文件）

【依赖与外部调用】
· LLM（异步，单次调用超时 60s，无重试）
· Milvus 向量检索（同步）
· BM25 本地索引

【已知约束】
· Bearer Token 鉴权
· REQUEST_TIMEOUT_SEC=300，MAX_PARAGRAPHS=200
· 无显式限流

【检查重点】
契约一致性 / 异常与边界 / 可观测性
```

### 示例 2 — 传统 CRUD 服务（Spring Boot + MyBatis）

```
【技术栈】
Java 17 + Spring Boot 3.2 + MyBatis-Plus + MySQL 8.0

【接口清单】
POST   /api/v1/orders              创建订单
GET    /api/v1/orders/{id}         查询订单详情
PUT    /api/v1/orders/{id}/cancel  取消订单（要求幂等）
GET    /api/v1/orders              分页查询订单列表

【路由代码】
@src/main/java/com/example/order/controller/OrderController.java

【请求 / 响应 Schema】
@src/main/java/com/example/order/dto/CreateOrderReq.java
@src/main/java/com/example/order/dto/OrderVO.java
@src/main/java/com/example/order/dto/PageReq.java

【依赖与外部调用】
· MySQL（同步，连接池 HikariCP，最大连接 50）
· Redis（缓存订单详情，TTL 300s）
· 库存服务 inventory-svc（HTTP，超时 2s，重试 1 次）
· 支付服务 payment-svc（HTTP，超时 5s，无重试）

【已知约束】
· JWT 鉴权（用户态），管理员需额外校验角色
· 单页 pageSize 上限 100
· 取消订单要求幂等（同一 orderId 多次取消只执行一次）
· SLA：P99 ≤ 300ms

【检查重点】
契约一致性 / 异常与边界 / 安全 / 性能
```

### 示例 3 — 文件上传接口（Node.js + Express）

```
【技术栈】
Node.js 20 + Express 4.19 + Multer 1.4 + TypeScript 5.3

【接口清单】
POST /upload/avatar    上传用户头像（图片，单文件）
POST /upload/batch     批量上传附件（最多 10 个）
GET  /files/{fileId}   下载文件

【路由代码】
@src/routes/upload.ts
@src/middlewares/auth.ts

【请求 / 响应 Schema】
multipart/form-data；响应使用 zod schema：
@src/schemas/upload.schema.ts

【依赖与外部调用】
· 对象存储（S3 兼容，异步 putObject，超时 10s）

【已知约束】
· Cookie Session 鉴权
· 单文件 ≤ 5MB；批量总大小 ≤ 30MB
· 仅允许 image/jpeg、image/png、application/pdf
· 文件名需做路径穿越防护

【检查重点】
安全 / 异常与边界 / 契约一致性
```

## 已知坑

1. **步骤 4 不会自动执行**：AI 容易在步骤 3 输出后直接续写修复方案。收到步骤 3 后，需主动说"先停这里"再做挑选。

2. **Schema 文件必须单独 `@`**：路由文件超过 200 行时，不单独 @ Schema 文件会导致 AI 误判请求 / 响应字段类型。

3. **疑似 / 已确认分级是关键**：不分级会导致修复方案被低置信度问题误导；若 AI 混淆两者，在回复中明确纠正即可。

4. **LLM / RAG 接口需补充重试策略**：在【依赖与外部调用】里注明重试策略与 token 计费方式，否则步骤 2 的"异常与边界"分析会遗漏这类非传统 Web 关注点。

5. **填写【已知约束】时勿粘贴真实 token**：Prompt 中已要求输出脱敏，但输入侧同样需要避免将真实鉴权凭据写入对话。
